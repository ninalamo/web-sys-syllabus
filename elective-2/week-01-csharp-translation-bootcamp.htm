1. The Hook & The Analogy

[SPEAK] Script: "You already know .NET concepts through VB. C# is the same ideas, different syntax. Think of it like speaking the same language with a different accent. Let's see how they compare — side by side."

[BOARD] Whiteboard Analogy: The Automatic vs. Manual Car

  • VB (Automatic): It handles the shifting for you (End If, End Sub).
  • C# (Manual): You have direct control over the gears. You declare types explicitly, close your own curly braces {}, and define exact access modifiers. Same road, different driving experience.

2. Core Concepts & Module Discussion

Topic A: Case Sensitivity (The #1 Bug Creator)

Module Syllabus Content: Unlike Visual Basic (VB), which is case-insensitive, C# is a strictly case-sensitive language. This means that the compiler treats uppercase and lowercase letters as completely different entities. For example, Console, console, and CONSOLE represent three entirely distinct identifiers in C#.

Because of this strictness, the C# ecosystem relies heavily on standardized naming conventions to keep code readable:

  • PascalCase (capitalizing the first letter of every word) is used for Classes, Methods, and Properties (e.g., WriteLine, StudentManager).
  • camelCase (lowercase first word, capitalized subsequent words) is used for local variables and method parameters (e.g., firstName, studentAge).

Failing to match the exact casing will result in a compiler error, specifically CS0103: The name does not exist in the current context.

[SPEAK] Instructor Script: "VB doesn't care if you yell at it or whisper. C# is extremely strict. If you miscapitalize a single letter, the compiler will crash."

[CODE] Live Example:

// WRONG: Throws CS0103
console.writeline("Hello"); 

// RIGHT
Console.WriteLine("Hello");

[TIP] Instructor Tip: Teach them the rule: Classes and Methods are PascalCase. Variables are camelCase. Force them to fix casing errors manually rather than relying on Auto-Correct.

Topic B: Statement Terminators (The Semicolon)

Module Syllabus Content: In VB, the end of a line of code (pressing Enter) tells the compiler that the statement is finished. In C#, whitespace, line breaks, and tabs are completely ignored by the compiler. Instead, C# requires an explicit statement terminator: the semicolon (;).

Because whitespace is ignored, you have the flexibility to write multiple statements on a single line, or stretch a complex statement across multiple lines for readability. However, forgetting to place a semicolon at the end of a logical statement is the most common syntax error for developers transitioning to C#, resulting in a CS1002: ; expected error.

[SPEAK] Instructor Script: "In VB, pressing Enter ends the line. In C#, pressing Enter does nothing. You must tell the compiler you are finished speaking by using a semicolon."

[CODE] Live Example:

var name = "Alice" // ERROR: Missing semicolon
var age = 25;      // SUCCESS

[TIP] Gen-Z Hook: The semicolon is the period at the end of a text message. It means "I'm done making my point."

Topic C: Scope Definition (Curly Braces)

Module Syllabus Content: Visual Basic defines the start and end of code blocks using verbose word pairs, such as If...End If, Sub...End Sub, or For...Next. C# replaces all of these verbose terminators with curly braces {}.

The opening brace { marks the beginning of a scope block, and the closing brace } marks the end. This applies to namespaces, classes, methods, loops, and conditional statements. While this makes C# code more concise, it requires developers to be diligent about indentation and formatting to visually track which brace closes which block.

[SPEAK] Instructor Script: "Instead of writing End If or End Sub, C# uses curly braces {} to create a visual box around a block of code."

[CODE] Live Example:

if (isAdmin) 
{
    Console.WriteLine("Welcome back!");
    // The brace below replaces 'End If'
}

[TIP] Instructor Tip: Tell students to ALWAYS format their document (Shift + Alt + F in VS Code) to line up their curly braces. A misaligned brace is the fastest way to break a program.

Topic D: Implicit Typing (var)

Module Syllabus Content: In older versions of C#, developers had to explicitly declare the type on both sides of an assignment (e.g., List<string> names = new List<string>();). C# introduced the var keyword to reduce this redundancy.

It is crucial to understand that var does not mean the variable is untyped or dynamic (like in JavaScript). It simply instructs the compiler to infer the data type based on the value on the right side of the equals sign during compilation. Once the type is inferred, it is locked in forever. You cannot assign a string to a variable that was inferred as an integer.

[SPEAK] Instructor Script: "This is the most misunderstood feature. var is NOT typeless like JavaScript. It just means 'Compiler, figure out the type based on the right side of the equals sign, and lock it in forever.'"

[CODE] Live Example:

var age = 25; // Compiler locks this in as an 'int'
age = "hello"; // ERROR: Cannot implicitly convert type 'string' to 'int'

[TIP] Instructor Tip: Show them the error live. Prove that var is strictly typed by hovering over the word var in the IDE to show that the tooltip reads int.

3. Code Walkthrough / Live Coding Blueprint

[SPEAK] Script: "Let me show you a class you already know in VB. Now here it is in C#. Don't read it — scan it. What's the same? What's different?"

  • Step 1: The "Hello World" Translation
    • Action: Write a simple VB Console output, then translate to C# Top-Level Statements.
  • Step 2: Translating a Class
    • Action: Show the transition from Public Property Name As String to public string Name { get; set; }.
  • Step 3: Translating an Array & Loop
    • Action: Translate a For Each loop into a C# foreach loop.

4. From the Trenches (Pro-Tip)

[TRENCHES] Instructor Script: "Why did the industry switch from VB to C#? Because C# is strictly enforced. When you're working on a codebase with 50 other developers, you don't want 'loose' code. You want strict rules so that if someone breaks something, the compiler yells at them immediately. C# forces you to be disciplined, which makes you a professional."

5. AI Integration & Debugging

  • AI Policy: Allowed: "In C#, how do I write a For Each loop?" Not Allowed: "Write my FizzBuzz exercise for me."
  • Common Error 1: CS1002 — ; expected. -> You forgot a semicolon. Check the end of your line.
  • Common Error 2: CS0103 — The name doesn't exist in the current context. -> 99% of the time, this is a case sensitivity issue. You typed console instead of Console.
